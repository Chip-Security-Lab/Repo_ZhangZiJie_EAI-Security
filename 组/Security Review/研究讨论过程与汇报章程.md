# 研究讨论过程与 PPT 汇报章程

---

## 一、讨论演进脉络

### 阶段 1：综述分析——建立问题全景

**做了什么：** 逐层深入分析 *Safety in Embodied AI: A Survey of Risks, Attacks, and Defenses*（2026，400+ 引文），梳理五层攻击面：

- §2 感知层：视觉/听觉/空间/运动/跨模态融合
- §3 认知层：指令理解/世界模型/推理
- §4 规划层：任务规划/轨迹规划/多智能体
- §5 执行交互层：控制/人机交互/多智能体协同
- §6 Agentic 层：工具使用/记忆/自我进化/级联风险

**关键产出：** 识别出具身AI最具特色的攻击方向——**语义-物理鸿沟**（BADROBOT 三种越狱攻击路径）、跨模态融合反直觉脆弱性、Grounding Failure（推理逻辑自洽但物理不可行）。每个方向都标注了"🔥等级"，区分传统攻击与具身独有攻击。

**过程中的重要讨论：**
- NeRF/3DGS 攻击 vs 世界模型攻击的区别——一个篡改"AI 看到的静态地图"（感知），一个篡改"AI 理解的物理规律"（认知）
- 跨模态融合中"单通道攻陷整体融合"的反直觉发现——融合不是冗余保护，融合机制本身就是攻击面
- BADROBOT 三种越狱手段如何具体构造——Contextual Jailbreak / Safety Misalignment / Conceptual Deception

---

### 阶段 2：AI 顶会扫描——确认问题空间

**做了什么：** 从 GitHub 合集 [Embodied-AI-Paper-TopConf](https://github.com/Songwxuan/Embodied-AI-Paper-TopConf) 中筛选 ~580 篇论文的安全相关研究。

**发现：** 仅 23 篇（~4%）与安全相关。集中在 VLA 鲁棒性（8 篇）和安全规划/控制（5 篇）。攻击类仅 2 篇（对抗+后门各 1），越狱/人机信任/记忆投毒全部为零。

**关键洞察：** AI 顶会有问题定义（VLA/LLM 安全），但攻击视角极窄。没有一篇做物理上下文安全校验。

---

### 阶段 3：安全四大调研——寻找方法基础

**做了什么：** 完整调研 IEEE S&P、CCS、NDSS、USENIX Security 2025-2026 共七个会议 ~1,500 篇论文。

**发现：** 安全四大几乎不做具身AI（直接相关不足 15 篇），但有大量可嫁接的技术储备：

- 物理域/传感器攻击：~42 篇（证明了物理传感器数据不可信）
- TEE/机密计算：~20 篇（"芯片防护"的学术基础）
- CPS/工控/车辆安全：~15 篇（架构同构——"指令到物理执行"链路）
- 硬件安全/侧信道：~60+ 篇（了解芯片能做什么）
- 固件/IoT/嵌入式：~38 篇（机器人的底层计算平台）

**最相关的单篇发现：**
- **TAT: Attesting Trajectory Integrity of Industrial Robotic Arms**（USENIX '26）——安全四大中唯一直接做机械臂轨迹完整性证明的论文
- **ControlLoc: Physical-World Hijacking Attack on Camera-based Perception in Autonomous Driving**（CCS 2025）
- **ConTest: Taming the Cyber-physical Input Space in Fuzz Testing with Control Theory**（CCS 2025）
- **TensorShield: Shielding Critical DNN Tensors with TEE**（CCS 2025）——软硬结合的范例

**关键洞察：AI 顶会定义了问题空间，安全四大提供了方法基础，但两边从未交叉。我们的贡献空间就在这个交叉地带。**

---

### 阶段 4：研究问题提炼——从大问题到可操作问题

**起点：** "具身指令的物理上下文安全校验"——太大了，需要三次收窄。

**第一次收窄——限定威胁类型：**
- 放弃防御传感器攻击（独立问题，安全四大有大量研究）
- 聚焦防御 **LLM 规划器被越狱** 后输出"语义正确但物理危险"的指令
- 攻击场景示例：被越狱的 LLM 输出"以最短路径到达目标"→ 经过湿滑厨房斜坡 → 负重 5kg → 摩擦力不足 → 摔倒。LLM 文本层面完全合理，物理层面致命

**第二次收窄——限定物理维度：**
- 不做所有物理上下文
- 聚焦 **地面稳定性**（摩擦系数、倾斜角、承重能力）
- 选择理由：传感器数据获取最直接（IMU+力传感器+深度相机），物理规律最清楚（摩擦公式+重心稳定性），安全阈值最容易量化
- 世界模型方向被**否定**：世界模型正确性验证是开放理论难题；我们的问题可用显式物理公式解决，不需要神经网络学物理

**第三次收窄——限定场景与任务：**
- 场景：家用移动操作机器人
- 任务："把 X 搬到 Y"类搬运任务（排除复杂接触式操控）
- 理由：移动搬运的物理安全性最可量化评估

**收窄后的研究问题：**
> 当机器人收到搬运类自然语言指令时，如何确保即使 LLM planner 被越狱，物理安全校验仍然可靠执行？

---

### 阶段 5：为什么是具身独有——威胁模型论证

**核心论证链条：**

1. **传统机器人中规划器是可信的**（A*/RRT/MPC——数学确定、不会"说谎"）
   → 安全校验的定位是辅助性冗余（防数值误差）
   → 软件层校验即可

2. **具身AI 中 LLM 规划器不可信**（可被语义越狱，BADROBOT 证明三种攻击路径）
   → 攻击者用自然语言"说服" LLM 输出危险指令
   → 这在传统机器人中不存在（无自然语言接口，无法"说服"规划器）
   → 这在纯 LLM 中后果不同（文本 ≠ 物理伤害）

3. **因为规划器不可信，安全校验的定位变了**
   → 从"辅助冗余"变成"最后防线"
   → 从"软件即可"变成"必须硬件隔离"（同 OS 的校验可被越狱的 LLM 绕过）

**结论：** 这是具身AI独有的安全威胁——"LLM 的语义越狱面 × 物理执行的后果严重性"。传统机器人安全和 LLM 安全都不覆盖这个问题。

---

### 阶段 6：TEE 定位的讨论与校准

**关键讨论——"TEE 是不是核心贡献"：**

用户提出质疑：TEE 是现成工具，我们不能改它，核心贡献应该是安全校验本身怎么做，TEE 只是锦上添花。

**讨论结论：**
- **核心贡献（科学问题）：** 语义-物理交叉安全校验方法本身——如何从传感器数据中提取物理参数、如何将指令语义映射到物理约束、如何设计安全判据
- **架构增强（工程贡献）：** TEE 隔离部署——保证校验逻辑不被攻陷的 OS 绕过
- TEE 是**拿来用的**（类比"我们用 Linux"），不是拿来改的
- 汇报时明确：80% 讲安全校验思路，20% 提 TEE 作为防御纵深

**TEE 的传统用途 vs 我们的新用途：**
- 传统：保护数据机密性（指纹不出安全区、密钥不泄露）
- 我们：保护物理安全判据的正确执行（判据不被绕过、不被篡改）

---

### 阶段 7：方案架构讨论

**三件事为什么不能全丢进 TEE：**
- TEE 资源极度受限（1-32MB 内存，无 Python/PyTorch/驱动）
- 设计目标就是"最少代码 = 最小攻击面"
- 软件侧做资源密集的事情（传感器处理），TEE 只跑几十行安全判据

**软件侧为什么用规则而非 AI：**
- "不能让 LLM 审查 LLM 的输出"——越狱的 LLM 不能自己审核自己
- 指令解析用简单规则（模板匹配/正则），物理判据用显式公式（摩擦力方程）

**为什么篡改传感器数据只会让判据更严格：**
- 攻击者篡改物理参数 → 只会让参数看起来更危险 → 判据更严格地拦截
- Attack surface 的单向性——安全设计上的好性质

---

### 阶段 8：可行性与路线

**纯软件学生能不能做 TEE：**
- 能。OP-TEE + QEMU 是纯软件仿真环境
- 写的是 C 代码，不需要硬件知识
- 安全判据代码不超过 500 行

**推进路线：**
1. 仿真阶段（2-3 月）：Gazebo 中验证物理上下文提取 + 安全判据
2. TEE 阶段（2 月）：QEMU + OP-TEE 中将判据迁移到 Secure World
3. 真机阶段（后期）：迁移到 PuppyPi 的树莓派 TrustZone

## 三、PPT 讲述要点备忘录

### P3（调研数据）

- ~1,500 篇论文中，具身/机器人直接相关不足 15 篇
- 但 TEE/硬件安全 ~60+ 篇，CPS/物理安全 ~42 篇——这些是现成的方法库
- 安全四大**没有**做具身AI安全，但**有**可嫁接的技术

### P5（三次收窄）

- 第一刀：不防传感器攻击（独立问题），防 LLM 越狱
- 第二刀：限定地面稳定性（数据最直接、公式最清楚）
- 第三刀：家用搬运场景（物理安全性最可量化）
- 世界模型被否定：我们的问题可用显式物理公式，不需要训练神经网络学物理

### P7（攻击场景）

- 被越狱的 LLM 输出"以最短路径到达目标"
- 经过湿滑厨房斜坡 + 负重 5kg → 摩擦力不足 → 摔倒
- 文本层面完全合理，物理层面致命——这就是语义-物理鸿沟

### P8（威胁模型变化）

- 核心对比表：传统机器人规划器可信 vs LLM 规划器不可信
- 由此推导：安全校验从"辅助冗余"变为"最后防线"

### P10（TEE 定位）

- 传统用 TEE 保护"秘密"（指纹、密钥不泄露）
- 我们用它保护"正确性"（安全判据不被篡改）
- 类比：用 Linux 是基础设施选择，不是科学贡献；TEE 同理
- 明确：我们是"用"TEE，不是"改"TEE

### P12（可行性）

- OP-TEE + QEMU 纯软件仿真，一条命令启动
- 校验逻辑不超过 500 行 C 代码
- 下阶段重点：安全判据的具体设计
