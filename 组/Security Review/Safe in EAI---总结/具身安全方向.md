# 具身智能安全研究方向探讨

> 基于 *Safety in Embodied AI: A Survey of Risks, Attacks, and Defenses* (2026) 的系统分析，以及顶会具身AI论文（ICML/NeurIPS/ICLR/CVPR/ICCV/CoRL/RSS，~580篇）的安全方向筛选，对可行研究方向进行收窄。

---

## 一、原始研究动机

两个观察引出研究兴趣：

1. **物理环境上下文变化导致安全属性翻转**：机器人在平地上执行的安全动作，换到斜坡/湿滑地面/狭窄空间后可能变成危险动作。机器人不理解"同一个动作在不同物理环境中安全属性不同"。

2. **无条件信任人类指令**：人机交互中，人类给出的指令可能是善意但物理上不安全的（人忽略了某些物理约束）。机器人缺乏对指令在当前环境下物理安全性的校验能力。

---

## 二、定位：当前研究的覆盖与空白

### 2.1 现有相关工作的分布

两个 idea 跨多层：

| 层 | 涉及到的子领域 | 现有研究状态 |
|----|-------------|------------|
| 感知层 §2 | 空间感知（环境几何/物理属性识别） | 较成熟，但以检测为主，不做安全推理 |
| 认知层 §3 | 接地失效（§3.3）、世界模型（§3.2） | 接地失效刚起步，世界模型安全还在概念验证阶段 |
| 规划层 §4 | 任务规划越狱（§4.1）、轨迹规划安全（§4.2） | 越狱攻击有概念框架（BADROBOT），安全规划以避障/力控为主 |
| 执行层 §5 | 人机交互信任（§5.2）、机器人控制安全（§5.1） | PsySafe 关注恶意信任操纵，不关注善意但危险的指令 |

### 2.2 关键空白

| 现有研究做的 | 没人做但你的 idea 指向的 |
|------------|---------------------|
| 检测指令是否**恶意**（越狱检测 §3.1） | 判断**善意**指令在当前物理环境下是否安全 |
| 物理避障/力控（传统机器人安全 §5.1） | 语义理解物理安全：不是"会不会撞"，而是"地面摩擦力够不够" |
| 人机交互中的**恶意**信任操纵（PsySafe §5.2） | 人机交互中**善意但无知**的指令——人类没想害机器人，但忽略的物理约束 |
| 纯文本空间的安全对齐（LLM safety） | **物理上下文感知**的安全对齐——同一句话在不同物理环境下安全性不同 |

### 2.3 顶会数据印证

- 具身AI安全在顶会中占比仅约 **4%**（23/580）
- 攻击类论文仅 **2 篇**（对抗 + 后门各 1），越狱攻击 **0 篇**
- 安全研究集中在 VLA 鲁棒性和安全规划/控制（共占约 57%）
- **物理语义交叉安全校验方向：顶会中零篇**

## 三、收窄方向 A：具身指令的物理上下文安全校验

### 3.1 研究问题

给定一条自然语言指令 + 当前物理环境状态，判断这条指令在当前环境下执行是否**物理上安全**。

### 3.2 核心洞察

现有安全研究要么做纯物理层（避障、力控——传统机器人），要么做纯语义层（越狱检测、安全对齐——LLM safety）。**物理语义的交叉校验目前无人做。**

```
纯语义层（LLM安全对齐）
    "推倒货架" ← 文本判断为危险 → 拒绝
    "结构完整性测试" ← 文本判断为安全 → 通过
                              ↓
                        物理执行 = 推倒货架
                              ↓
纯物理层（机器人控制安全）
    障碍物检测、力限制、急停 ← 但这些都不拦截"测试"动作本身

═══════════════════════════════════════
你的方向：物理语义交叉层
    输入：指令（语义）+ 环境参数（物理）
    判断：这条指令在当前具体物理环境下是否安全
    输出：安全/不安全/需确认
    这是纯语义层和纯物理层之间的空白地带
═══════════════════════════════════════
```

### 3.3 具体场景示例

| 人类指令 | 物理环境 | 现有系统行为 | 你的系统应该做什么 |
|---------|---------|------------|-----------------|
| "把箱子搬到那边去" | 地面倾斜 15°，摩擦系数低 | 规划路径 → 搬箱子走过去 → 滑倒 | 识别斜坡 + 计算"负重走斜坡的安全性" → 拒绝/要求确认 |
| "把热水倒进杯子里" | 杯子是塑料的，水温 95°C | 执行倒水 → 杯子变形 → 溢出 | 比对杯子耐温 vs 水温 → 提示风险 → 建议换玻璃杯 |
| "从上面取下那个盒子" | 盒子在高处，下面是不平稳的椅子 | 用椅子垫脚 → 获取盒子 | 判断椅子不稳定 → 拒绝 → 建议使用梯子 |
| "把剩余的化学品倒进下水道" | 化学品标签显示"禁排"，但LLM未知 | 执行倾倒 → 污染 | 读取标签 → 与物理安全数据库交叉比对 → 拦截 |

### 3.4 需要什么

- VLA 模型（如 RT-2, Octo, OpenVLA）或 LLM+VLM 组合架构
- 物理仿真环境（SafeLab、LIBERO、Isaac Sim）
- 环境物理参数的提取能力（从传感器数据中估计摩擦系数、倾斜度、温度等）
- 安全知识库（物体物理属性、安全阈值）

### 3.5 可用的基础设施

| 资源 | 用途 | 来源 |
|------|------|------|
| SafeLab | 科学实验机器人安全测试环境 | ICML 2026 |
| LIBERO-Plus | VLA 鲁棒性分析基准 | CVPR 2026 |
| EmbodiedBench | 多模态agent综合评估框架 | ICML 2025 |
| Agent-SafetyBench | agent安全评估基准 | 2025 |
| SafeAgentBench | 感知→执行全链路安全基准 | 2025 |

### 3.6 定位

属于综述子方向：§3.3 接地失效 + §5.1 安全控制 的交叉。

与现有工作的关系：

| 相关工作 | 与你的区别 |
|---------|----------|
| Grounding Failure (§3.3) | 接地失效是**被动失败**（AI"将就"了错误方案）。你主动做**安全校验** |
| BADROBOT Safety Misalignment (§4.1) | BADROBOT 关注恶意利用语义-物理鸿沟。你关注善意但物理无知的情况 |
| PsySafe (§5.2) | PsySafe 关注恶意人格投毒和信任操纵。你的场景里没有恶意方 |
| SafeDec (ICML 2026) | SafeDec 是轨迹层约束解码（"不能撞"）。你需要语义层约束（"这种情况下不该做这个任务"） |

## 四、收窄方向 B：人机指令交互中的安全质疑机制

### 4.1 研究问题

机器人应该在什么时候、以什么方式**主动反问他人类**来验证指令的物理安全性？

### 4.2 核心洞察

现有安全交互假设两种极端：要么"人类说的都对"（过度信任 → 危险），要么"拒绝执行"（信任不足 → 机器人不好用）。你提出的第三条路：**不拒绝也不盲从，而是反问验证**。

### 4.3 具体场景

```
人类："把箱子搬到那个角落去"
机器人（分析环境）→ 发现角落有裸露电线
机器人反问："那个角落有电线，如果我搬箱子过去可能碰到它，
            你确认要我搬过去吗？还是先关掉电源？"
人类："哦对，先关掉电源。"

——这是"盲从导致事故"被一次安全质疑拦截的过程。
```

### 4.4 额外挑战（相对于方向 A）

- 需要做人机对话系统的设计与评估
- 涉及人因实验设计
- 安全质疑的"频率"需要校准——太频繁让人烦躁、太少不安全

## 五、方向对比与建议

| 维度 | 方向 A：物理上下文安全校验 | 方向 B：安全质疑机制 |
|------|:--:|:--:|
| 研究边界 | 清晰——一个判断任务 | 较宽——涉及对话策略和HRI |
| 是否可直接用现有基准做实验 | 是（SafeLab、LIBERO-Plus） | 需自建或扩展基准 |
| 是否需要人类受试实验 | 不需要 | 需要 |
| 顶会空白程度 | 直接空白 | 间接空白（PsySafe 相关但不重叠） |
| 与已有工作的边界 | 清晰（见表 3.6） | 需要更多厘清 |
| 实现周期 | 较短（模型+仿真即可） | 较长（需要HRI实验） |
| **建议优先级** | ⭐⭐⭐ | ⭐⭐ |

**建议从方向 A 切入**：任务定义清晰、可利用现有基准、不需要人因实验、与现有工作的边界明确。方向 B 可作为方向 A 的自然延伸——在安全校验的基础上，加上人机交互的质疑与确认机制。

## 六、下一步

1. 确定方向后，进一步缩小范围到具体任务（如：家庭场景下的安全操控 / 仓库场景下的安全导航 / 实验室场景下的安全实验操作）
2. 选定一个具体任务，收集或构建环境物理参数数据
3. 在现有 VLA 模型上做安全校验模块的 prototype
4. 与现有安全对齐方法做对比（LLM safety guardrail vs 你提出的物理上下文感知 guardrail）
